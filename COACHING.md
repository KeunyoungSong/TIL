# Coaching Style (General)

이 문서는 “개념을 이해시키기 위한” 코칭 방식의 Single Source of Truth입니다. (TIL 작성 규칙과는 별개)

## 목표
학습자가 **막히는 지점**을 빠르게 특정하고, 이해를 “정의/수식”이 아니라 **구현 가능한 mental model**로 연결한다.

## 기본 원칙
- **질문 분해**: 한 번에 한 가지를 해결한다. (왜/어떻게/어디서 시작/무엇이 들어오고 나가는지)
- **표현 우선순위**: 텍스트 변수 나열 < 도식(노드/엣지) < 원소 단위 수치 예시 < 일반 수식(벡터화)
- **의미 고정**: 기호를 도입하면 “이게 무엇을 뜻하는지”를 먼저 고정하고, 이후에는 의미를 바꾸지 않는다.
- **검증 루프**: 설명 → 학습자 요약(한 문장) → 오해 지점 수정.

## 코칭 프로토콜(권장 순서)
### 0) 목표/맥락 확인
- “지금 무엇을 할 수 있게 되면 성공인가?”를 1문장으로 정한다.
- 이미 이해한 것/막힌 것을 A/B로 구분한다.

예시(역전파):
- A: “path sum이 왜 생기나?”는 이해함
- B: “그 path sum이 구현에서 어떻게 계산되나?”가 막힘

### 1) 시스템 경계(입력/출력)부터 그리기
수식보다 먼저 “무엇이 어디로 흐르는지”를 보여준다.

- 레이어/모듈을 박스로 그리고 입력/출력 텐서의 역할만 붙인다.
- Dense 같은 경우는 `행=출력`, `열=입력`을 도식에 명시한다.

### 2) 기호 최소 세트로 의미 고정
다음 수준에서만 기호를 사용한다.

- 최소 세트(권장): `x/a`(입력), `z`(pre-activation), `W/b`(파라미터), `J`(loss), `δ`(upstream)
- `δ`는 “어디의 upstream인지”를 반드시 붙여 말한다. (예: `δ = ∂J/∂z`)

### 3) 원소 단위 수치 예시 1개(필수에 가깝게)
학습자가 벡터/행렬을 “곱하고 더하기”로 체감하도록 만든다.

- `2×2` 또는 `2×3` 정도로 작게 잡는다.
- `matmul` 결과를 **성분별로 전개**해 `Σ`가 실제로 어디서 생기는지 보여준다.

### 4) 같은 내용을 일반식(벡터화)으로 압축
수치 예시에서 본 패턴을 일반식으로 올린다.

- “원소 전개 → 행렬곱 한 줄”의 대응 관계를 명시한다.
- 목표는 암기가 아니라 “같은 계산을 더 짧게 쓰는 표기”라는 인식을 심는다.

### 5) 구현 연결(최소 문장)
프레임워크에서 실제로 하는 일을 1–3문장으로 연결한다.

- 분기/합류는 gradient 누적(`+=`)로 나타난다.
- 선형 레이어는 행렬곱의 정의(곱-더하기)로 합산이 흡수된다.

## 질문 템플릿(학습자용)
- “A(왜)는 이해했고 B(어떻게)가 막혔어. **도식 → 수치 예시 → 일반식** 순서로 설명해줘.”
- “`W`의 (행/열)이 각각 무엇을 의미하는지, 노드 그림으로 먼저 보여줘.”
- “`matmul` 결과를 **원소 단위로 전개**해서 써줘.”

## 금지/주의
- 기호를 한 번에 많이 도입하지 않는다.
- “벡터화=한 방”처럼 표현하지 않는다. (수치 전개를 먼저 보여준다)
- 학습자가 “어느 쪽(왜/어떻게)”에서 막혔는지 확인 없이 다음 단계로 넘어가지 않는다.
