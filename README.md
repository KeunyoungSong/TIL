# TIL (Today I Learned)

간단한 “Today I Learned” 기록 저장소입니다.

## 규칙
- 배운 내용 하나당 파일 하나.
- 파일 이름은 작성 날짜로 시작.
- 파일 이름/문서 제목은 기본 한글, 기술 용어/기술 스킬은 영어 사용.
- **Troubleshooting 타입**은 제목만 봐도 환경/문제가 드러나도록 쓰기(예: `Kaggle`, `Python 3.11`, `scikit-learn`, `설치 실패` 등). 해결 방안까지 제목에 넣을 필요는 없음.
- **Concept 타입**은 개념/주제가 바로 드러나게 쓰기(예: `Gradient Descent 기본 개념`, `CSS Grid 기초`). 필요하면 간단한 범위/맥락을 덧붙이기.
- 짧고 실용적으로 작성.
- 수식은 GitHub 렌더링 호환을 위해 `$...$` / `$$...$$`를 사용하고, `$$...$$` 블록 수식은 리스트 항목 안에 넣지 않는다(본문에 독립시키고 앞뒤로 한 줄 띄우기).
- (권장) Concept 글에는 “당위/배경”을 위해 `역사 타임라인(Timeline)` 또는 `왜 그렇게 됐나(Why/Constraints)`를 3–6줄로 추가한다.
  - 예: “언제/누가/무엇이 표준이 됐는지”, “당시 제약(컴퓨팅/데이터/이론/구현)이 무엇이었는지”
  - 근거가 있으면 `참고(Links)`에 같이 남긴다.
- (권장) 수식/벡터화가 들어가는 Concept 글은 `원리/수식(Principle/Math)` 앞에 `기호 정리(Symbols)`를 둬서 기호 의미를 먼저 정리한다.
- (권장) 복잡한 흐름은 이미지 대신 텍스트 도식(ASCII)이나 짧은 단계(1–4)로 먼저 설명한 뒤 수식으로 내려간다.

## 문서 타입(Type)
이 저장소는 아래 2가지 타입만 관리합니다. 문서 제목(H1) 바로 아래에 타입을 한 줄로 표기합니다.

- `> Type: Troubleshooting`
  - 특정 환경/문제에서 발생한 실패를 원인 분석 → 해결 과정 → 교훈 중심으로 정리.
  - `TEMPLATE.md`의 Troubleshooting 섹션 흐름을 따른다.
- `> Type: Concept`
  - 개념/원리를 배경 → 정의 → 핵심 아이디어 → 예시/주의점 중심으로 정리.
  - `TEMPLATE.md`의 Concept 섹션 흐름을 따른다.
  - (선택) “설명”보다 “회고” 톤으로 쓰고 싶으면 `TEMPLATE.md`의 `회고형 블록(Reflective)` 선택 섹션을 사용한다.

## 파일 이름 형식
`YYYY-MM-DD-제목.md` (기술 용어는 영어)

예시:
- `2025-12-12-Redis-파이프라인.md`
- `2025-12-13-CSS-Grid-기초.md`

## 작성 방법
1. `TEMPLATE.md`를 복사해 새 파일을 만듭니다.
2. 빠르게 채웁니다.
3. 작성/수정 후 먼저 검토를 요청합니다.
4. 승인받으면 그때 커밋하고 바로 `push`합니다.

인덱스는 만들지 않습니다. 파일명 검색으로 찾습니다.
