## Troubleshooting

<!-- 아래는 문제 해결 기록용 흐름 -->

## 문제(Problem)
무슨 문제가 있었는지, 에러/증상을 구체적으로 적기.
여러 문제가 있으면 번호로 나누기(예: 1), 2)).

## 원인(Cause)
왜 그런지 한 문장으로 요약하고, 근거/맥락을 덧붙이기.
문제 번호에 대응해서 원인을 적으면 흐름이 선명해짐.

## 해결(Solution)
실제로 한 조치들을 순서대로 적기.
`문제(Problem)`의 번호에 맞춰 “어떤 조치가 어떤 문제를 해결했는지”를 대응해서 기록.
우회/대안을 쓰는 경우에는 왜 그 방법을 채택했는지(승인/제약/트레이드오프)를 함께 적기.
```bash
명령어가 있으면 여기에
```
```python
코드가 있으면 여기에
```

## 결과/현재 상태(Result/Status)
각 문제가 해결됐는지, 부분 해결인지, 미해결인지와 남은 제약을 한 줄씩 정리.
예:
- 문제 1) 설치 실패 → 최신 버전 설치로 해결
- 문제 2) `load_boston` ImportError → API 제거로 미해결, 원본 데이터 직접 로드로 대체

## 배운 점(Takeaways)
다음에 같은 상황이 오면 어떻게 할지, 핵심 교훈 2–4줄.

## 정의(Definition)
한 문장으로 개념을 요약하고, 범위/대상을 명확히 적기.

## 핵심 아이디어(Key Ideas)
왜 필요한지, 어떤 직관으로 동작하는지 3–5줄.

## 예시(Examples)
간단한 예시나 상황 적용. 코드/수식/그림 중 편한 형태로.
```python
예시 코드가 있으면 여기에
```

## 주의할 점(Caveats)
헷갈리기 쉬운 포인트, 자주 하는 실수, 경계 조건.

## 한 줄 요약(Summary)
나중에 다시 봤을 때 바로 떠오를 문장 하나.

## 내가 먼저 고정한 한 문장(결론)
헷갈림을 끝내준 문장 하나.

## 내가 헷갈렸던 이유
어떤 표현/용어/가정 때문에 꼬였는지 2–5줄.

## 내가 머릿속에서 쓰는 그림
이미지 대신 텍스트 도식(ASCII), shape, 짧은 예시.

## 내가 정한 기준
무엇을 어떻게 세는지/구분하는지(예: “필터 기준” vs “2D 조각 기준”).

## 복구용 체크 질문
헷갈리면 되돌아갈 질문 1–3개.
